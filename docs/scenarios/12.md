# Scenario 12: Pipeline Task Monitoring and Observability

> Pipeline tasks SHALL be monitor-able and observable

## Implementation

Every step execution produces a `StepTrace` with name, status, and real wall-clock duration. The full execution yields an `ExecutionResult` containing all traces and artifacts.

### Step Traces (`src/thingfactory/types.gleam:125-143`)

- `StepTrace(step_name, status, duration_ms)` — per-step execution record
- `StepStatus`: `StepOk | StepFailed(StepError) | StepSkipped`
- `ExecutionResult(result, trace, artifacts)` — complete run observable

### Real Timing (`src/thingfactory/timing.gleam`)

- `timing.measure(fn)` — wraps execution with `Date.now()` (JS) / `erlang:system_time` (Erlang)
- Wall-clock milliseconds, not CPU time

### Progress Events (`src/thingfactory/types.gleam:146-158`)

- `StepStarting(name, index, total)` — emitted before step runs
- `StepFinished(name, index, total, status, duration_ms)` — emitted after
- Used by CLI compact/verbose modes and GUI real-time updates

### Database Persistence (`web/db/migrations/`)

- `step_traces` table stores name, status, duration_ms, error_msg per step
- `pipeline_runs` table tracks overall run status, duration, timestamps
- GUI Gantt timeline visualizes step timing from stored traces
