# Scenario 65: Pluggable Executor Models

**Specification:** The system SHALL support various executor models. By default,
the local docker execution model should be used to run pipelines. However, the
system should be designed to support other execution models such as kubernetes,
remote execution, or even serverless functions. The execution model should be
pluggable and configurable by the user.

## Implementation

### `web/lib/types.ts` — Executor type hierarchy

```typescript
export type ExecutorKind = "local" | "docker";

export interface LocalExecutorConfig {
  kind: "local";
}

export interface DockerExecutorConfig {
  kind: "docker";
  image: string; // Docker image to run commands in
  volumes?: string[]; // Extra volume mounts (host:container)
}

export type ExecutorConfig = LocalExecutorConfig | DockerExecutorConfig;
```

### `web/lib/trigger-pipeline.ts` — Executor dispatch

Three layers implement the pluggable model:

1. **`executeLocal`** — runs `sh -c <command>` in the current working directory.
2. **`executeDocker`** — runs
   `docker run --rm -v <workingDir>:/workspace -w /workspace <image> sh -c <command>`.
3. **`executeCommand`** — dispatch switch on `executor.kind`, exhaustive with
   `never` guard for future kinds.

```typescript
function executeCommand(
  input: ExecInput,
  executor: ExecutorConfig,
): Promise<CommandResult> {
  switch (executor.kind) {
    case "docker":
      return executeDocker({ ...input, dockerCfg: executor });
    case "local":
      return executeLocal(input);
    default: {
      const _exhaustive: never = executor;
      return executeLocal(input);
    }
  }
}
```

### `web/db/migrations/0008_executor_config.sql`

Adds the `executor` column to `pipeline_definitions` (default `local`):

```sql
ALTER TABLE pipeline_definitions ADD COLUMN executor TEXT NOT NULL DEFAULT '{"kind":"local"}';
```

The in-memory schema in `web/lib/db.ts` was updated in parallel.
