# Scenario 47: Push/Pull Pipeline Model

## Status: WORK_COMPLETE

## Requirement

> Pipelines SHALL be easy to express work in both an imperative (PUSH model) as
> well as workers in the pipeline PULLing work from a queue (PULL model). This
> shall be enabled by the substrate, but because it's a common pattern, pipeline
> tools SHOULD be considered for offering this to users as a first class
> feature.

## Implementation

### Substrate: Message Store

`src/thingfactory/message_store.gleam` provides the foundational substrate
enabling both push and pull patterns. Steps publish messages to topics and other
steps retrieve them.

### First-Class Queue API

`src/thingfactory/work_queue.gleam` — convenience module built on the message
bus:

- `enqueue(ctx, queue_name, item)` — add work items to a named queue
- `pull_all(ctx, queue_name)` — pull all pending items from a queue
- `has_work(ctx, queue_name)` — check if a queue has items
- `queue_size(ctx, queue_name)` — get item count

Queues are namespaced with a `queue:` prefix to avoid collisions with regular
pub-sub topics.

### Example Pipeline

`src/thingfactory/examples.gleam:queue_worker_pipeline()` demonstrates the PULL
model:

1. **produce_work** — enqueues 3 compilation tasks to "tasks" queue
2. **worker** — pulls all items from queue and processes them
3. **summarize** — reports results

Runnable via CLI: `gleam run -m thingfactory/cli -- run queue`

### Both Models Demonstrated

- **PUSH**: every existing pipeline (step output flows to next step input via
  function arguments)
- **PULL**: `queue_worker_pipeline` (worker pulls items from a named queue via
  `work_queue.pull_all`)
