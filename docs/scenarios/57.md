# Scenario 57 — Gitea Repository Integration in GUI

## Requirement

> The user SHALL be able to select a gitea instance and repository to connect to
> in the GUI, and select an autopopulated branch from a dropdown to register for
> Pipelines.

## Implementation

### Database

Migration `web/db/migrations/0007_gitea_connections.sql` adds:

```sql
CREATE TABLE IF NOT EXISTS gitea_connections (
  id          TEXT PRIMARY KEY,
  url         TEXT NOT NULL,
  token       TEXT NOT NULL,
  repo        TEXT NOT NULL,
  branch      TEXT NOT NULL,
  pipeline_id TEXT,
  created_at  TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))
);
```

`web/lib/db.ts` adds `GiteaConnectionsTable` interface (with `url`, `token`,
`repo`, `branch`, `pipeline_id`, `created_at`) and includes it in
`ThingfactoryDB`. The in-memory test schema is also updated.

### Gitea API Proxy Routes

Four server-side routes proxy the Gitea REST API using
`Authorization: token <token>`:

| Route                                       | Purpose                                                          |
| ------------------------------------------- | ---------------------------------------------------------------- |
| `GET /api/gitea/repos?url=&token=`          | Lists repos via `GET /api/v1/repos/search`                       |
| `GET /api/gitea/branches?url=&token=&repo=` | Lists branches via `GET /api/v1/repos/{owner}/{repo}/branches`   |
| `GET /api/gitea/connections`                | Lists saved connections (token excluded)                         |
| `POST /api/gitea/connections`               | Saves a connection; optionally sets pipeline trigger to `Gitea:` |
| `DELETE /api/gitea/connections/[id]`        | Removes a connection                                             |

When a connection is saved with a `pipeline_id`, the pipeline's trigger is set
to `{ Gitea: { repo: "owner/repo", events: ["push"] } }`.

### UI — `/integrations` Page

`web/app/integrations/page.tsx` now includes a **Connect Gitea Repository**
section below the GitHub section, with cascading dropdowns:

1. **Instance URL** input — e.g. `https://gitea.example.com`
2. **Token input** (`type="password"`) + "Load Repositories" button — calls
   `GET /api/gitea/repos`
3. **Repository dropdown** — autopopulated; shows `owner/repo` full names with
   `[private]` indicator
4. **Branch dropdown** — autopopulated when repo is selected; pre-selects
   `default_branch`; calls `GET /api/gitea/branches`
5. **Pipeline selector** — optional; links the connection to an existing
   pipeline
6. **Register Connection** button — POSTs to `/api/gitea/connections`

A **Gitea Connections** card lists registered connections with instance URL,
repo, branch, linked pipeline, timestamp, and a Remove button.

### Test Infrastructure

`web/lib/seed-fixtures.ts` `clearAllData()` now deletes `gitea_connections` rows
before all other tables for proper test isolation.

### Tests

`web/e2e/integrations-gitea.spec.ts` — 7 passing E2E tests:

- Page loads and shows Gitea form (URL input, token input, load button)
- Empty state shown when no Gitea connections exist
- Cascading dropdowns: repo → branch (Playwright route mocks for Gitea API)
- Branch auto-selection to repo's `default_branch`
- Branch count matches mock data
- Pipeline selector and Register button appear after branch selection
- Full registration flow: connection appears in list after save
- Delete flow: connection removed from list; empty state restored

All 15 integrations E2E tests pass (7 Gitea + 8 GitHub, no regressions).
