# Scenario 59: GitHub & Gitea Webhook Events Activate Triggers

> The system SHALL make events from GitHub & Gittea work so as to activate triggers.

## Implementation

### Webhook Receiver Endpoints

Two new API routes receive push events from external VCS providers and trigger linked pipelines:

**`POST /api/webhooks/github`** — `web/app/api/webhooks/github/route.ts`
- Accepts standard GitHub push event payload: `{ ref, repository: { name, owner: { login } } }`
- Extracts branch by stripping `refs/heads/` prefix from `ref`
- Queries `github_connections` for rows matching `org + repo + branch` with a non-null `pipeline_id`
- Triggers each matched pipeline via the shared `triggerPipeline` utility
- Returns `{ ok, triggered, run_ids }` (200) or 400 if required fields are missing

**`POST /api/webhooks/gitea`** — `web/app/api/webhooks/gitea/route.ts`
- Accepts Gitea push event payload: `{ ref, repository: { full_name } }`
- Extracts branch the same way
- Queries `gitea_connections` for rows matching `repo + branch` with a non-null `pipeline_id`
- Triggers matched pipelines identically

### Shared Pipeline Triggering — `web/lib/trigger-pipeline.ts`

Extracted the pipeline run creation and async step execution logic that was previously duplicated into a single reusable `triggerPipeline(pipelineId, triggerType)` function. The existing manual trigger route (`/api/pipelines/[name]/[version]/trigger/route.ts`) was refactored to use it, eliminating duplication (DRY).

### TypeScript Type — `web/lib/types.ts`

Added `Gitea` variant to the `TriggerConfig` union type:

```typescript
| { Gitea: { repo: string; events: string[] } }
```

This aligns with the `GitHub` variant already present and matches what `POST /api/gitea/connections` writes to the database.

### Connection → Trigger Flow

1. User registers a connection via the Integrations page (scenarios 56/57), selecting a repo, branch, and pipeline.
2. The connection is stored in `github_connections` / `gitea_connections` with `pipeline_id`.
3. When a push event arrives at the webhook endpoint, matching connections are found and `triggerPipeline` fires a new run with `trigger_type: "webhook"`.
