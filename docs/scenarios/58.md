# Scenario 58: In-Code Trigger Specification

> The pipeline SHALL allow in code specifying execution on key trigger, such as
> `Trigger.BranchUpdate("main")` or `Trigger.Schedule("0 0 * * *")`.

## Status: VERIFIED

## Implementation

### `BranchUpdate` Trigger (`src/thingfactory/webhook_trigger.gleam`)

`BranchUpdate(branch: String)` is a first-class variant of the `Trigger` type:

```gleam
pub type Trigger {
  NoTrigger
  WebhookTrigger(matcher: WebhookMatcher)
  ManualTrigger
  BranchUpdate(branch: String)   // ← new
}
```

Pipelines opt in via the builder API:

```gleam
pipeline.new("ci", "1.0.0")
|> pipeline.with_trigger(BranchUpdate("main"))
```

or via the convenience helper:

```gleam
|> pipeline.with_trigger(webhook_trigger.on_branch_update("main"))
```

### `BranchMatcher` (`src/thingfactory/webhook_trigger.gleam`)

A `BranchMatcher(branch: String)` variant was added to `WebhookMatcher` for use
inside composite `WebhookTrigger` matchers:

```gleam
WebhookTrigger(AllMatcher([SourceMatcher("github"), BranchMatcher("main")]))
```

### `branch` field on `WebhookEvent`

`WebhookEvent` now carries `branch: Option(String)`. Push-event helpers set it
automatically:

| Helper                                      | `branch` value |
| ------------------------------------------- | -------------- |
| `github_push_event(repo, branch, …)`        | `Some(branch)` |
| `gitlab_push_event(project, branch, …)`     | `Some(branch)` |
| `gitea_push_event(repo, branch, …)` _(new)_ | `Some(branch)` |
| `github_pr_event(…)`                        | `None`         |
| `custom_event(…)`                           | `None`         |

### `Schedule` trigger

Schedule-based execution is specified separately via `pipeline.with_schedule/2`
using the `Cron(expression)` variant (e.g. `Cron("0 0 * * *")`), verified under
scenario 15. Both approaches coexist: a pipeline can carry both a `trigger` and
a `schedule`.
