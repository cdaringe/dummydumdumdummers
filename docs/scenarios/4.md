# Scenario 4: Type-Safe Gleam Pipeline Definitions

> Pipelines SHALL be buildable via typesafe gleam code

## Implementation

Pipelines are defined using a type-safe builder API in Gleam. The `Pipeline(output)` type uses a phantom type parameter to track the final output type at compile time.

### Builder API (`src/thingfactory/pipeline.gleam`)

- `pipeline.new(name, version)` creates a `Pipeline(Nil)`
- `pipeline.add_step(name, fn)` appends a step and returns `Pipeline(Dynamic)`
- `pipeline.add_step_with_deps(name, fn, deps)` adds a step with DAG dependencies
- `pipeline.add_step_with_loop(name, fn, loop)` adds a step with loop configuration
- `pipeline.with_timeout()`, `with_schedule()`, `with_trigger()`, `with_secrets()` configure pipeline-level settings

Each step function has a typed signature: `fn(Context, Dynamic) -> Result(Dynamic, StepError)`. The Gleam compiler enforces these signatures at call sites. Intermediate types are erased to `Dynamic` between steps (documented trade-off).

### Core Types (`src/thingfactory/types.gleam`)

- `PipelineId(name, version)` — unique pipeline identity
- `StepError` — `StepFailure | StepTimeout | ArtifactNotFound` (no silent failures)
- `PipelineError` — `StepError | LoadError | ValidationError`
- `Context` — artifact store, message bus, dependency bindings, secrets
- `ExecutionResult(a)` — result + trace + artifacts

### Key Design Decisions

- Pipeline type is public (not opaque) to enable test mocking
- Steps stored internally as type-erased `fn(Context, Dynamic)` for flexibility
- Gleam compiler catches type mismatches at compile time, not runtime
