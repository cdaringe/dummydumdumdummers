# Scenario 28: No String Typing

> The pipeline tasks SHALL NOT be stringly typed or referenced. Stringly typing
> in general is to be minimized.

## Implementation

Gleam's type system eliminates stringly-typed references throughout the pipeline
system:

- **Pipeline IDs** ŌĆö `PipelineId(name, version)` record type, not bare strings
- **Step errors** ŌĆö `StepError` sum type:
  `StepFailure | StepTimeout | ArtifactNotFound`
- **Pipeline errors** ŌĆö `PipelineError` sum type:
  `StepError | LoadError | ValidationError`
- **Schedules** ŌĆö `Schedule` sum type with 6 variants (not cron strings as
  primary)
- **Triggers** ŌĆö `Trigger` and `WebhookMatcher` sum types with typed matchers
- **Loops** ŌĆö `Loop` sum type: `FixedCount | RetryOnFailure | UntilSuccess`
- **Context** ŌĆö typed `Dict(String, Dynamic)` with `get_dep()` returning
  `Result`
- **Step IDs** ŌĆö `Pipeline(id, output)` type parameter; `pipeline.typed()` uses
  user-defined enum types for step identifiers

### Typed Step ID Dependencies

Step-to-step dependencies use the `id` type parameter directly. With
`pipeline.typed()`, step IDs are user-defined enum values ŌĆö the Gleam compiler
rejects unknown or misspelled step identifiers at build time:

```gleam
type BuildStep { Checkout Lint Test Build }

pipeline.typed("parallel_build", "1.0.0", step_to_string)
|> pipeline.add_step_with_deps(Checkout, checkout_fn, [])
|> pipeline.add_step_with_deps(Lint, lint_fn, [Checkout])
|> pipeline.add_step_with_deps(Test, test_fn, [Checkout])
|> pipeline.add_step_with_deps(Build, build_fn, [Lint, Test])
```

The `depends_on` list holds values of type `id` ŌĆö no string coercion possible.

String-ID pipelines via `pipeline.new()` still work but bypass the compile-time
identity check (string literals can be misspelled). The typed API is the
recommended approach for new pipelines.
