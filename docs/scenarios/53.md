# Scenario 53: CLI Dockerfile with Erlang Target

## Requirement

> The CLI Dockerfile SHALL be gleam based using erlang target.

## Implementation

The root `Dockerfile` has been updated to build and run the Gleam CLI using the
**Erlang/OTP target** instead of the JavaScript target.

### Build Stage

Uses `ghcr.io/gleam-lang/gleam:v1.14.0-erlang-alpine` — Gleam's official image
bundling both Gleam and Erlang/OTP:

```dockerfile
FROM ghcr.io/gleam-lang/gleam:v1.14.0-erlang-alpine AS builder
WORKDIR /app
COPY manifest.toml gleam.toml ./
RUN gleam deps download
COPY src src
RUN gleam export erlang-shipment
```

`gleam export erlang-shipment` compiles all Gleam source to `.beam` files and
packages them with an `entrypoint.sh` script into `./build/erlang-shipment/`.

### Runtime Stage

Uses `erlang:27-alpine` — a minimal Erlang/OTP runtime (no Gleam toolchain
required):

```dockerfile
FROM erlang:27-alpine
RUN apk add --no-cache docker-cli
WORKDIR /app
COPY --from=builder /app/build/erlang-shipment .
ENTRYPOINT ["./entrypoint.sh", "run", "thingfactory", "main"]
```

The `entrypoint.sh` (generated by `gleam export erlang-shipment`) starts the
BEAM VM with correct `-pa` code paths and invokes `thingfactory:main/0`, which
dispatches to the Gleam CLI entry point (`src/thingfactory/cli.gleam`).

Docker run arguments are forwarded as plain Erlang arguments accessible via the
`argv` library (backed by `init:get_plain_arguments/0` on Erlang).

### Erlang FFI

All Erlang FFI modules are compiled as part of the shipment:

- `src/thingfactory/thingfactory_erlang_cli.erl` — pipeline loading, file I/O,
  cwd
- `src/thingfactory/timing_erlang.erl` — wall-clock timing
- `src/thingfactory/runner_host_erlang.erl` — CPU core detection
- `src/thingfactory/command_runner_erlang.erl` — shell command execution

### Key File

- `Dockerfile` — root CLI Dockerfile, now Gleam/Erlang based
