# Scenario 43: Self-Hosting with Docker

> The project SHALL be excellent to self host. Hobbyists SHOULD feel easy launching a container with docker-in-docker.

## Implementation

### Docker Deployment

- `Dockerfile` — multi-stage build; runtime stage installs `docker-cli` (`apk add --no-cache docker-cli`) so the container can spawn pipeline containers via the host Docker socket.
- `docker-compose.yml` — one-command deployment; CLI service mounts `/var/run/docker.sock` for socket-based Docker access (no privileged DinD daemon required).
- `DOCKER.md` — step-by-step Docker and self-hosting documentation.

### Ease of Use

```bash
# Start web GUI (persistent SQLite database on port 3000)
docker-compose up -d web

# Run pipeline with Docker isolation from inside the CLI container
docker-compose run --rm cli run --isolator docker thingfactory@examples:basic_pipeline

# Run pipeline with local isolation (no Docker spawn)
docker-compose run --rm cli run --isolator local thingfactory@examples:basic_pipeline
```

### Docker-Socket Approach

The CLI container accesses the host Docker daemon via the mounted socket — the industry-standard sibling-container approach. No `--privileged` mode or nested Docker daemon required. The host's Docker daemon spawns the pipeline containers.

| Requirement | How It Is Met |
|---|---|
| Docker CLI in runtime image | `apk add --no-cache docker-cli` in `Dockerfile` runtime stage |
| Socket access for CLI service | `- /var/run/docker.sock:/var/run/docker.sock` in `docker-compose.yml` |
| Hobbyist self-hosting docs | `DOCKER.md` Self-Hosting section + `docs/HOSTING_SERVICE.md` |
