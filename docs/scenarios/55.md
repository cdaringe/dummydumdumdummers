# Scenario 55: GUI Node State During Pipeline Run

## Requirement

> The GUI SHALL show the node state when a pipeline is running, such as "pending", "running", "failed", "succeeded", etc. It should also show the duration of each node, and the overall pipeline in the flow diagram itself.

## Implementation

### Flow Diagram on Run Detail Page

The run detail page (`web/components/run/RunDetailClient.tsx`) now includes a **live Flow Diagram** section that renders the pipeline DAG with real-time state and duration for each node.

Key changes:
- **`web/lib/dag.ts`**: Added `duration_ms?: number` to `StepNodeData` interface; updated `buildDagGraph()` to accept `TraceData` (status + optional duration per step).
- **`web/components/pipeline/PipelineDagNode.tsx`**: Each node now displays the step duration alongside status. Running nodes show an animated pulse indicator.
- **`web/components/run/RunDetailClient.tsx`**: Added a "Flow Diagram" card above the Gantt timeline that builds a live `DagGraph` from `allSteps` (which merges step definitions with live SSE trace data). The diagram updates as SSE events (`step_started`, `step_completed`) arrive.
- **`web/app/pipelines/[name]/[version]/page.tsx`**: Updated to pass `{ status, duration_ms }` objects to `buildDagGraph()` from the last-run traces.

### Node States Shown

| Status | Color | Description |
|--------|-------|-------------|
| `pending` | Gray | Step not yet started |
| `running` | Green (pulsing) | Step actively executing |
| `ok` | Green | Step completed successfully |
| `failed` | Dark | Step failed |

### Duration Display

Each node in the flow diagram shows the step duration (e.g. `1.2s`, `340ms`) when the step has completed. Overall pipeline duration is shown in the run summary card.

### Real-Time Updates

The flow diagram is driven by the same SSE stream (`/api/runs/[runId]/stream`) as the step traces list, ensuring the DAG updates live while the pipeline is running.

### DB Proxy Fix (Bonus)

Also fixed a pre-existing bug introduced in recent commits where the Kysely `db` proxy incorrectly bound functions, causing `db.fn.count` to fail (breaking dashboard, runs list, and stats pages). The proxy now uses a nested Proxy for function values that:
1. Applies method calls with the correct `this` (the Kysely instance, needed for private `#props` field access)
2. Preserves own properties on function objects (e.g. `db.fn.count`, `db.fn.avg`)

See `web/lib/db.ts` — `wrapValue()` helper.
