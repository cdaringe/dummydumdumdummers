# Scenario 67: Executor Capability Labels

**Specification:** The system SHALL support pipelines opting in to run on
executors with certain labels or capabilities (e.g. please run this pipeline on
an agent with GPU capabilities). This allows pipeline authors to specify only
minimum requirements and the system schedules the pipeline on any available
executor that meets them.

## Implementation

### `web/lib/types.ts` — New types

Three additions:

1. **`LabeledExecutorRequirement`** — stored in `pipeline_definitions.executor`
   when a pipeline uses label-based selection instead of a pinned executor kind:
   ```typescript
   export interface LabeledExecutorRequirement {
     kind: "labeled";
     requiredLabels: string[];
   }
   ```

2. **`PipelineExecutor`** — discriminated union of what the executor column can
   hold:
   ```typescript
   export type PipelineExecutor = ExecutorConfig | LabeledExecutorRequirement;
   ```

3. **`ExecutorInstance`** — a named executor registered in the service pool:
   ```typescript
   export interface ExecutorInstance {
     id: string;
     labels: string[]; // e.g. ["docker", "gpu", "linux"]
     config: ExecutorConfig; // concrete executor to use when selected
   }
   ```

   `labels?: string[]` was also added to `LocalExecutorConfig` and
   `DockerExecutorConfig` so individual executor configs can advertise their
   capabilities.

### `web/lib/config.ts` — `executorPool` field

`ServiceConfig` gains `executorPool: ExecutorInstance[]`, populated from the
`THINGFACTORY_EXECUTOR_POOL` environment variable (parsed as JSON). When the
variable is absent the default pool is used:

```typescript
function defaultExecutorPool(): ExecutorInstance[] {
  return [
    { id: "default", labels: ["local", "standard"], config: { kind: "local" } },
  ];
}
```

| Variable                     | Default     | Description                              |
| ---------------------------- | ----------- | ---------------------------------------- |
| `THINGFACTORY_EXECUTOR_POOL` | (see above) | JSON array of `ExecutorInstance` objects |

### `web/lib/trigger-pipeline.ts` — Label matching & resolution

**`findMatchingExecutor`** returns the first pool entry whose label set is a
superset of the required labels:

```typescript
function findMatchingExecutor(
  requirement: LabeledExecutorRequirement,
  pool: ExecutorInstance[],
): ExecutorConfig | null {
  const match = pool.find((instance) =>
    requirement.requiredLabels.every((label) => instance.labels.includes(label))
  );
  return match?.config ?? null;
}
```

**`triggerPipeline`** resolves the pipeline executor before creating a run:

```typescript
const pipelineExecutor: PipelineExecutor = JSON.parse(pipeline.executor ?? '{"kind":"local"}');

let executor: ExecutorConfig;
if (pipelineExecutor.kind === "labeled") {
  const resolved = findMatchingExecutor(pipelineExecutor, config.executorPool);
  if (!resolved) {
    throw new Error(`No executor in the pool satisfies labels: [${...}]`);
  }
  executor = resolved;
} else {
  executor = pipelineExecutor;
}
// Allow-list check applies to both resolved and direct executors
```

If no pool instance satisfies the requirements the error is caught by the
trigger route and returned as HTTP 403.

## Verification

- Types in `types.ts` provide compile-time safety for all three new concepts.
- `config.ts` parses `THINGFACTORY_EXECUTOR_POOL` from env; falls back to a
  single default local executor labelled `["local", "standard"]`.
- `trigger-pipeline.ts` resolves labeled requirements at trigger time; throws a
  descriptive error listing available executors when no match is found.
- The allow-list check (scenario 68) still applies after label resolution.
- 88 E2E tests pass with no regressions (2 new tests added).
