# Scenario 66: Pipeline-Defined Execution Model

**Specification:** The system SHALL support pipelines defining their execution
model (e.g. please run this pipeline on a docker agent X).

## Implementation

### `web/db/migrations/0008_executor_config.sql`

Each `pipeline_definition` row stores its executor config as a JSON blob:

```sql
ALTER TABLE pipeline_definitions ADD COLUMN executor TEXT NOT NULL DEFAULT '{"kind":"local"}';
```

### `web/lib/db.ts` — Kysely schema

```typescript
interface PipelineDefinitionsTable {
  executor: Generated<string>;   // JSON ExecutorConfig
  ...
}
```

### `web/lib/trigger-pipeline.ts` — Per-pipeline executor dispatch

`triggerPipeline` reads the pipeline's `executor` field from the database and
uses it to select the right executor for each step:

```typescript
const executor: ExecutorConfig = JSON.parse(
  pipeline.executor ?? '{"kind":"local"}',
);
// Passed to executeCommand() for every step in the run
void executeStepsInBackground({ runId, steps, executor });
```

### `web/lib/seed-fixtures.ts` — Example with explicit executor

The `thingfactory_deploy` pipeline demonstrates per-pipeline executor config:

```typescript
{
  name: "thingfactory_deploy",
  executor: { kind: "local" },
  ...
}
```

The `insertPipeline` statement in `seedFixtures` serialises it:

```typescript
JSON.stringify(p.executor ?? { kind: "local" });
```

## Verification

- `executor` column present in both the migration file and in-memory schema.
- `triggerPipeline` parses and enforces executor per pipeline.
- `thingfactory_deploy` fixture shows explicit per-pipeline executor selection.
