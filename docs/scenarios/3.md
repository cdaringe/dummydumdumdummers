# Scenario 3: Diataxis-Compliant Documentation

> The project shall offer documentation adherent to Diataxis, mainly "how to" and "reference". We value conciseness and linking to other resources over long-form documentation.

## Implementation

Documentation now has an explicit Diataxis entry point and a concise split between how-to and reference materials.

### Documentation Structure

| Guide | Diataxis Type | Purpose |
|---|---|---|
| `docs/README.md` | Navigation | Diataxis index grouping all user docs by type |
| `docs/GETTING_STARTED.md` | How-to | Install and run first pipeline |
| `docs/RUNNING_PIPELINES.md` | How-to | Run pipelines, modes, artifacts, Docker |
| `docs/HOSTING_SERVICE.md` | How-to | Deploy and operate the service |
| `docs/USER_GUIDE.md` | Reference | Concepts and architecture map |
| `docs/WEB_GUI_GUIDE.md` | Reference | GUI and API behavior reference |
| `docs/TROUBLESHOOTING.md` | Reference | Problem/solution catalog |

### Evidence

| Artifact | Description |
|---|---|
| `docs/README.md` | Diataxis index with how-to vs reference grouping and quick paths |
| `docs/GETTING_STARTED.md` | TOC + cross-links to deeper docs for next tasks |
| `docs/RUNNING_PIPELINES.md` | TOC + command reference and links from onboarding docs |
| `docs/HOSTING_SERVICE.md` | TOC + deployment reference with links to troubleshooting |
| `docs/USER_GUIDE.md` | TOC + architecture reference and links to task-oriented guides |
| `docs/WEB_GUI_GUIDE.md` | TOC + page/API reference for GUI users |
| `docs/TROUBLESHOOTING.md` | TOC + concise issue lookup with direct fixes |
| `README.md` | Top-level documentation section linking Diataxis index |
