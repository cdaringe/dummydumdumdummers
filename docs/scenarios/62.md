# Scenario 62: Import Pipeline Steps from a Library

## Requirement

> A user SHALL be able to import pipeline steps into a pipeline from another
> library and use them in conjunction with the local pipeline steps.

## Implementation

### How It Works

Pipeline steps in Thingfactory are ordinary Gleam functions with the signature:

```gleam
fn(Context, Dynamic) -> StepResult(Dynamic)
```

Any Gleam module—local or a third-party package—can export step factory
functions with this signature. They integrate with `pipeline.add_step` exactly
like inline lambdas, with full compile-time type checking across module
boundaries. Critically, library steps are **not coupled to any specific `id`
type**: the same function works whether the pipeline uses `String` identifiers
or a custom typed enum.

### Step Library Module

**`src/thingfactory/step_library.gleam`** exports three reusable step factories:

```gleam
pub fn validate_non_empty() -> fn(Context, Dynamic) -> StepResult(Dynamic)
pub fn to_uppercase()       -> fn(Context, Dynamic) -> StepResult(Dynamic)
pub fn prefix(label: String) -> fn(Context, Dynamic) -> StepResult(Dynamic)
```

### String-ID Example (`Pipeline(String, Dynamic)`)

**`src/thingfactory/examples.gleam`** — `library_import_pipeline()` interleaves
library steps with locally-defined steps using plain string step IDs:

```gleam
import thingfactory/step_library

pub fn library_import_pipeline() -> pipeline.Pipeline(String, Dynamic) {
  pipeline.new("library_import", "1.0.0")
  |> pipeline.add_step("seed",        fn(_ctx, _input) { Ok(dynamic.string("hello")) })
  |> pipeline.add_step("validate",    step_library.validate_non_empty())
  |> pipeline.add_step("local_enrich", fn(_ctx, input) { ... })
  |> pipeline.add_step("uppercase",   step_library.to_uppercase())
  |> pipeline.add_step("prefix",      step_library.prefix("RESULT: "))
}
```

### Typed-ID Example (`Pipeline(LibraryStepId, Dynamic)`)

**`src/thingfactory/examples.gleam`** — `library_import_typed_pipeline()` uses
the same library steps but with a typed enum for step IDs, providing
compile-time exhaustiveness checking and preventing typos:

```gleam
pub type LibraryStepId {
  LibSeed
  LibValidate
  LibEnrich
  LibUppercase
  LibPrefix
}

pub fn library_import_typed_pipeline() -> pipeline.Pipeline(LibraryStepId, Dynamic) {
  pipeline.new("library_import_typed", "1.0.0")
  |> pipeline.add_step(LibSeed,     fn(_ctx, _input) { Ok(dynamic.string("hello")) })
  |> pipeline.add_step(LibValidate, step_library.validate_non_empty())
  |> pipeline.add_step(LibEnrich,   fn(_ctx, input) { ... })
  |> pipeline.add_step(LibUppercase, step_library.to_uppercase())
  |> pipeline.add_step(LibPrefix,   step_library.prefix("RESULT: "))
}
```

Step names are automatically serialised from enum variants via
`pipeline.step_id_to_string` (uses `string.inspect` and strips surrounding
quotes for strings):

| Enum variant   | Serialised step name |
| -------------- | -------------------- |
| `LibSeed`      | `"LibSeed"`          |
| `LibValidate`  | `"LibValidate"`      |
| `LibEnrich`    | `"LibEnrich"`        |
| `LibUppercase` | `"LibUppercase"`     |
| `LibPrefix`    | `"LibPrefix"`        |

### Existing Library Examples

The same pattern is used by built-in modules:

| Module              | Step factories                                       |
| ------------------- | ---------------------------------------------------- |
| `command_runner`    | `step/2`, `step_in_dir/3`, `sh/1`, `sh_in_dir/2`    |
| `kubernetes_runner` | `step/3`                                             |
| `step_library`      | `validate_non_empty/0`, `to_uppercase/0`, `prefix/1` |

All are imported via standard Gleam `import` and used identically in
`pipeline.add_step`, compatible with any `id` type parameter.
