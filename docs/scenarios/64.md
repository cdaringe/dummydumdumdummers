# Scenario 64: Graceful Teardown and Restart

**Specification:** The system SHALL support a graceful teardown and restart.
Specifically, the system SHALL support blocking starting new pipelines, allowing
in-flight pipelines to complete, and then shutting down. It should also be able
to restart without loss of data or state, and SHALL RESUME pipelines that were
previously blocked on next boot.

## Implementation

### Drain / Shutdown Mode

**`web/lib/shutdown.ts`** — singleton drain-state manager stored in `globalThis`
(same pattern as `run-events.ts` to survive hot-reload).

| Export | Behaviour |
|---|---|
| `isShuttingDown()` | Returns `true` while drain is active |
| `registerActiveRun()` | Called when `executeStepsInBackground` starts |
| `unregisterActiveRun()` | Called in `finally` when execution finishes; resolves pending drain waiters when count hits 0 |
| `initiateGracefulShutdown()` | Sets `draining=true`, returns a `Promise<void>` that resolves once all active runs finish |
| `resetShutdownState()` | Resets to clean state (used by test reset and drain toggle) |
| `getActiveRunCount()` | Inspection helper |

### Blocking New Triggers

**`web/lib/trigger-pipeline.ts`** `triggerPipeline()`:

```typescript
if (isShuttingDown()) {
  // Insert run with status="blocked", return runId immediately — do NOT execute
  await db.insertInto("pipeline_runs").values({ ..., status: "blocked" }).execute();
  return runId;
}
```

`executeStepsInBackground` wraps execution with `registerActiveRun()` / `unregisterActiveRun()` in a `finally` block so the drain counter is always decremented.

### Startup Recovery

**`web/lib/startup.ts`** `resumeBlockedRuns()`:

1. Queries all runs with `status = "blocked"` and calls `resumeBlockedRun(id)` for each — transitions status to `"running"` and fires `executeStepsInBackground`.
2. Queries all orphaned `"running"` runs (from a prior crash) and marks them `"failed"` via `markOrphanedRunFailed()`.

### Server Instrumentation

**`web/instrumentation.ts`** — Next.js server-side hook (runs once per process):

```typescript
export async function register() {
  if (process.env.NEXT_RUNTIME === "nodejs") {
    const { nodeStartup } = await import("./lib/node-startup");
    await nodeStartup();
  }
}
```

**`web/lib/node-startup.ts`** — isolated to prevent Next.js edge-runtime static
analysis from flagging `process.once`:

- Registers `SIGTERM` / `SIGINT` handlers → calls `initiateGracefulShutdown()` → waits for in-flight runs → `process.exit(0)`.
- Calls `resumeBlockedRuns()` on every boot to replay queued runs and clean up orphans.

### Admin API Endpoints

| Route | Method | Purpose |
|---|---|---|
| `/api/admin/drain` | `POST` | Enter drain mode (block new triggers) |
| `/api/admin/drain` | `DELETE` | Exit drain mode |
| `/api/admin/resume-blocked` | `POST` | Resume all blocked runs (mirrors boot recovery) |

### Test Reset Integration

`POST /api/test/reset` calls `resetShutdownState()` so every E2E test starts
with drain mode off, regardless of what a previous test did.

### UI

`web/components/ui/StatusBadge.tsx` — added `blocked: "Blocked"` to the label
map so runs with `status="blocked"` display a styled "Blocked" badge.

## Verification

4 E2E tests in `web/e2e/graceful-shutdown.spec.ts` (all passing):

1. **Drain blocks triggers** — `POST /api/admin/drain` → trigger pipeline →
   `GET /api/runs/:id` returns `status: "blocked"`.
2. **Exit drain allows triggers** — enter then exit drain → trigger pipeline →
   run is not `"blocked"`.
3. **Resume-blocked re-executes** — drain → trigger (blocked) → exit drain →
   `POST /api/admin/resume-blocked` → run transitions away from `"blocked"`.
4. **No-op resume** — with no blocked/running runs, resume-blocked returns `{ok: true}`.

All 92 E2E tests pass; no regressions introduced.
