# Scenario 60 — Root Pipeline File for Self-Building

## Requirement

> The project SHALL host a pipeline file in the root such that a deployed
> instance of the project can build itself.

## Status: VERIFIED

## Implementation

### Root-Accessible Pipeline File

`src/thingfactory/build.gleam` is the canonical, root-accessible pipeline file
for building Thingfactory. It is designed to be the first file a developer or CI
operator would point to:

```bash
thingfactory run -f src/thingfactory/build.gleam build
```

The `-f` flag causes the Erlang-target CLI to:

1. Derive the module name from the file path (`thingfactory@build`)
2. Walk up to find `gleam.toml` and run `gleam build --target erlang` if needed
3. Load and invoke the `build` function via Erlang's `apply/3`

### Pipeline Structure

The `build()` function defines a parallel DAG pipeline:

| Step              | Depends On                 | Command                       |
| ----------------- | -------------------------- | ----------------------------- |
| `gleam_check`     | —                          | `gleam check`                 |
| `gleam_format`    | —                          | `gleam format --check`        |
| `gleam_build_erl` | check, format              | `gleam build --target erlang` |
| `web_install`     | —                          | `npm --prefix web install`    |
| `web_build`       | web_install                | `npm --prefix web run build`  |
| `verify`          | gleam_build_erl, web_build | inline assertion              |

Steps without dependencies run in parallel via
`parallel_executor.execute_parallel/3`.

### Key Files

- **`src/thingfactory/build.gleam`** — The root pipeline file (new)
- **`src/thingfactory/parallel_executor.gleam`** — DAG-aware executor used by
  `run()`
- **`src/thingfactory/command_runner.gleam`** — `step/2` wraps shell commands as
  pipeline steps
- **`src/thingfactory/thingfactory_erlang_cli.erl`** —
  `load_pipeline_from_file/2` resolves, builds, and loads the file at runtime

### Differentiation from Scenario 40

Scenario 40 (`dogfood_pipeline` in `examples.gleam`) demonstrates the dogfooding
concept as one of 20+ examples. Scenario 60 requires a **dedicated, top-level
file** that a deployed instance uses as its canonical self-build entry point.
`build.gleam` fulfills this: it is standalone, well-documented, and directly
named for its purpose.

## Verification

```bash
gleam check   # zero errors, zero warnings
```

The file compiles cleanly. On an Erlang-target build:

```bash
thingfactory run -f src/thingfactory/build.gleam build
```
